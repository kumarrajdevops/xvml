XVML vs HTML vs Markdown

Head-to-head analysis against two real production websites

Benchmarked 2026-06-13 against live fetches of news.ycombinator.com and github.com/login. Each page was recreated in all three formats with identical real content, then measured for source size, capability coverage, and output quality. This report itself is written in XVML.

Executive summary

90%↑Confidence: app-style pages
85%↑Confidence: AI generation
55%→Confidence: long-form prose
25%↓Confidence: custom design

XVML is a high-confidence choice inside its niche — forms, dashboards, settings pages, docs portals, internal tools, and AI-generated UI — where it is 2–6× more compact than hand-written HTML and does things Markdown structurally cannot. It is not a replacement for HTML when pixel-perfect custom design is the goal, nor for Markdown when the content is long-form prose.

Methodology

  1. Fetched two live production pages: the Hacker News front page (content/list archetype) and the GitHub login page (app/form archetype)
  2. Measured the real pages: raw HTML bytes, tag counts, external asset dependencies
  3. Recreated each page in XVML, hand-written HTML (fair: minimal, semantic, inline CSS), and Markdown — using the real fetched content (actual story titles, points, authors)
  4. Rendered the XVML versions with xvml render; all six artifacts measured with wc
  5. Counted LLM tokens for every artifact with gpt-tokenizer (cl100k_base) — a standard, reproducible proxy; Claude's tokenizer differs slightly but preserves the ratios
  6. Scored capability coverage and per-use-case confidence from the measured evidence

Scenario A — Hacker News front page (content/list archetype)

A1 · XVML — 963 bytes, 23 lines

Data-driven: the six stories live in @data state, the row template is an @each. Changing the stories means editing JSON, not markup — or fetching it live with @data src=/stories.json.

xvml
@spec 1
@page "Hacker News" light
@nav Hacker News | new | past | comments | ask | show | jobs | submit
@data
{
  "stories": [
    { "rank": 1, "title": "CRISPR tech selectively shreds cancer cells…", "points": 323, "by": "signa11" },
    { "rank": 2, "title": "I Am Not a Reverse Centaur", "points": 47, "by": "doctorow" },
    { "rank": 3, "title": "A PDF that changes based on how its read", "points": 64, "by": "pdfgeek" }
  ]
}
# close the @data block with @-@-end (two @ signs)
@card
  @each story in stories
    @text "{story.rank}. {story.title}" bold
    @text "{story.points} points by {story.by}" muted
  @end
@end

A3 · Markdown — 646 bytes, 16 lines

The smallest source — Markdown's home turf is exactly this kind of linked list. But it needs an external converter plus a stylesheet to become a web page, the nav bar is just a row of links with no styling control, and there is no path to scores updating live.

B3 · Markdown — structurally impossible

✕Markdown has no form elements. No inputs, no buttons, no password masking, no checkboxes that submit. The closest approximation is a textual sketch.
md
# Sign in to GitHub
Username or email address: ____________
Password: ____________
**[ Sign in ]**
- [ ] Keep me signed in

This renders as prose with blanks. It is not a form and can never become one without abandoning Markdown for embedded raw HTML — at which point you are writing HTML.

Overall analysis

Token utilization (LLM cost of each format)

Tokens are the unit of cost, latency, and context budget when an AI writes or edits a page. Measured with cl100k_base on the exact artifacts above:

ArtifactXVMLHTMLMarkdown
HN front page — source323 tokens683 tokens218 tokens
Login page — source120 tokens832 tokens123 tokens (non-functional)
Rendered output (HN / login)7,228 / 5,556source is the outputn/a
Real production page (HN / GitHub login)—11,868 / 14,321 tokens—
6.9×↑fewer output tokens than HTML (login)
119×↑fewer than the real GitHub login page
2.1×↑fewer than HTML (HN page)
46×↑cheaper round-trip edits vs rendered HTML
  • Generation: an LLM producing the login page emits 120 tokens of XVML vs 832 of equivalent HTML — ~7× less output means ~7× lower cost and latency per page, since output tokens are billed at a premium and stream serially
  • Editing: to change a page the model must re-read it. Keeping XVML as the source of truth costs 120–323 tokens of context per page; using rendered HTML as the source costs 5,556–7,228 — a 17–46× context tax
  • Markdown is token-competitive with XVML on pure content (218 vs 323) but cannot express the form scenario at any token count
  • Error surface scales with output length: 120 constrained-grammar tokens validated by xvml check is a far smaller space for hallucination than 832+ tokens of free-form HTML

This is why xvml ask works reliably at temperature 0: the model writes the short, constrained representation, and the deterministic renderer expands it into the long one.

Where each format honestly wins

XVML wins

  • App-style pages: 2–6× less source than HTML
  • Zero styling decisions — consistent design system
  • Single-file deterministic output
  • Reactivity with no framework or build step
  • AI generation: constrained grammar, spec-checkable, temperature 0

HTML wins

  • Unconstrained design — any layout, any brand
  • Arbitrary JavaScript logic
  • Universal ecosystem and tooling
  • No abstraction ceiling — full platform access

Markdown wins

  • Long-form prose — lowest friction ever designed
  • Smallest source for pure content (646 B vs 963 B)
  • Ubiquitous support (GitHub, editors, chat)
  • Trivially diffable and reviewable

Known XVML limitations

MUSTdesign-ceiling

No custom CSS per element — the design system is the design. Brand-specific layouts are out of scope

MUSTlogic-ceiling

Actions are a fixed grammar (set/toggle/push/remove/fn) — arbitrary client logic requires fn: escape hatches to window functions

SHOULDoutput-weight

Self-containment costs bytes: every page carries the full CSS (~16 KB) even when it uses three commands

SHOULDecosystem

One implementation, one design system, no third-party component market — maturity risk vs 30 years of HTML

Final verdict

✓High confidence (85–90%) inside the niche: structured app-style pages, internal tools, docs portals, and AI-generated UI. The login benchmark — 386 bytes of XVML versus 2,487 bytes of hand-written HTML versus structural impossibility in Markdown — is the clearest single data point.
⚠Moderate confidence (55%) for prose-heavy content: Markdown remains the better authoring format for articles; XVML is viable when the prose needs to live inside a styled, navigable site.
✕Low confidence (25%) for pixel-perfect custom design: that is HTML's home ground and XVML does not compete there by design.

Positioning: XVML does not replace HTML or Markdown. It occupies the gap between them — pages too interactive for Markdown but too structured to justify hand-writing HTML. Within that gap, the measurements favour it consistently.
